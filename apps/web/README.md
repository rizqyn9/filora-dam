# Filora Web

React 19 SPA — the thin front-end for the Filora DAM API. All business logic
lives in `apps/api`; this app handles presentation, data fetching, and UI state.

## State management strategy

Two clearly-separated concerns — **do not mix them**:

- **Server state** → **TanStack Query**. Anything that comes from the API
  (galleries, assets, albums, the current user). Owns caching, refetching,
  mutations, invalidation. Defined per-feature in `features/<x>/api.ts`.
- **Client/UI state** → **Zustand** (`src/stores`). Only ephemeral or
  device-local UI: theme, sidebar, multi-select, and the auth token mirror.

If you're tempted to put API data in Zustand, use a Query hook instead.

## Getting started

```bash
bun install
cp .env.example .env      # adjust VITE_API_* if needed
bun run dev               # http://localhost:3000  (proxies /api → :8080)
```

Scripts: `dev`, `build`, `preview`, `typecheck`, `lint`, `lint:fix`,
`format`, `format:check`.

## Linting & formatting (Oxc)

- **oxlint** (`.oxlintrc.json`) and **oxfmt** (`.oxfmtrc.json`) — Rust-based,
  installed locally in this app.
- `bun run lint` / `bun run lint:fix`, `bun run format` / `bun run format:check`.

### VS Code in this monorepo

Both tools auto-discover the **nearest** config per file (nested config), so the
configs in `apps/web/` apply automatically even with the repo opened at the root.

> Do **not** set `oxc.configPath` / `oxc.fmt.configPath` in VS Code — that pins a
> single config and disables nested lookup (the classic "only reads root config"
> problem). Leave them empty. See `.vscode/settings.json` at the repo root.

Most reliable per-app setup: open **`filora.code-workspace`** (multi-root). The
Oxc server then treats `apps/web` as its own workspace folder, auto-detects the
local binaries, and applies `apps/web/.vscode/settings.json`.

### Why feature-first?

It mirrors the API's vertical-slice modules (`account`, `gallery`, `asset`, …).
Each feature owns its schemas, data hooks, and UI, so a feature can grow or be
removed without ripple effects. Shared infra stays in `lib/`; shared UI in
`components/`.

## Adding a feature (recipe)

1. `src/features/<name>/schemas.ts` — Zod schemas mirroring the API models
   (see `apps/api/API.md`); export inferred types.
2. `src/features/<name>/api.ts` — a `<name>Keys` factory, `queryOptions`
   builders, and `useX` query/mutation hooks calling `lib/api-client`.
3. `src/features/<name>/components/` — feature UI.
4. `src/routes/<name>/…` — routes; prefetch in `loader` via
   `context.queryClient.ensureQueryData(...)`.

## API integration

- `lib/api-client.ts` speaks the Filora envelope (`{ success, data | error }`),
  unwraps `data`, and throws `ApiError` on failure.
- **Auth is decoupled.** The client gets its bearer token from an injected
  provider. Today it's wired to `auth-store` (`initAuthBridge()` in `main.tsx`).
  To adopt Clerk, call `setAuthTokenProvider(() => clerk.session?.getToken())`
  at bootstrap — nothing else changes.
- Responses are validated with Zod at the feature boundary before entering the
  cache.

```bash
bunx --bun shadcn@latest add <component>
```
